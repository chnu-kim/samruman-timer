# 단위 테스트 실행

`npm test`를 실행하여 모든 단위 테스트 결과를 확인하고, 실패 시 원인 분석 및 수정 방안을 제시한다.

## 동작 순서

### 1단계: 테스트 실행

```bash
npm test
```

### 2단계: 결과 분석

테스트 결과를 파싱하여 아래 형식으로 출력한다:

```
## 🧪 단위 테스트 결과

| 파일 | 테스트 수 | 통과 | 실패 | 상태 |
|------|-----------|------|------|------|
| lib/db.test.ts | 3 | 3 | 0 | ✅ |
| ... | ... | ... | ... | ... |
| **전체** | **N** | **N** | **0** | ✅/❌ |
```

### 3단계: 실패 분석 (실패 시에만)

실패한 테스트가 있으면:

1. 에러 메시지와 스택트레이스를 읽는다
2. 관련 소스 파일을 Read로 확인한다
3. 원인을 분석하여 아래 형식으로 보고한다:

```
### ❌ 실패 분석

#### [테스트 파일명] > [테스트 이름]
- **에러**: 에러 메시지
- **원인**: 분석 결과
- **수정 방안**: 구체적 수정 제안 (파일 경로 + 변경 내용)
```

4. 사용자에게 자동 수정 여부를 묻는다

### 4단계: 커버리지 갭 확인

테스트가 모두 통과하면, 현재 테스트 파일 목록과 소스 파일 목록을 대조하여 테스트 미작성 파일을 식별한다.

**테스트 파일 위치 규칙:**
- `src/lib/*.ts` → `src/lib/__tests__/*.test.ts`
- `src/proxy.ts` → `src/__tests__/proxy.test.ts`
- `src/app/api/**/*.ts` → `src/__tests__/api/*.test.ts`

```
## 📊 커버리지 갭

| 소스 파일 | 테스트 파일 | 상태 |
|-----------|------------|------|
| src/lib/db.ts | src/lib/__tests__/db.test.ts | ✅ |
| src/app/api/projects/route.ts | src/__tests__/api/projects.test.ts | ✅ |
| src/app/api/timers/[id]/graph/route.ts | — | ⚠️ 미작성 |
```
